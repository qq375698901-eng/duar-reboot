### 当前已实现
- 角色基础状态机已落地：
  - 顶层状态：OPERABLE / STUN / LAUNCH / DOWN / GRABBED

## 2026-03-26 PVE 首只敌人建议
- 第一只正式小怪建议做“近战步兵”，不要先做盾卫、远程怪、突刺怪或机制怪。
- 设计定位：
  - 它不是教学木桩，而是“会主动压迫你、但规则清楚”的第一只敌人。
  - 它的职责是验证你当前这套人物手感，尤其是距离判断、前摇阅读、贪刀惩罚、受击反馈和小心推进的战斗节奏。
- 为什么它应该是第一只：
  - 你的项目当前最成熟的是近距离攻防、受击、倒地、守御反击和招式节奏；近战步兵能直接覆盖这些已完成系统。
  - 如果第一只就做盾卫或远程怪，会把测试重点带偏到“解机制”而不是“验证基础战斗循环”。
  - 如果第一只就做高速突刺怪，玩家很容易把压力理解成“反应考试”，不利于建立你想要的谨慎探索节奏。
  - 近战步兵最适合拿来校准“掉血是否心疼”“是否会因为贪刀被反打”“一对二时是否开始紧张”这三件核心体验。
- 第一只敌人的推荐原型：
  - 名称建议：地牢刀手 / 失乡佣兵 / 破甲囚徒，任选其一；当前推荐先用“地牢刀手”作为开发代号。
  - 武器形态：单手短刀或破旧直剑。
  - 体型：与玩家接近或略矮，避免首怪就引入过大的判定阅读成本。
  - 核心行为：缓慢接近 -> 进入中近距离试探 -> 单次挥砍 -> 短暂恢复 -> 继续压迫。
- 初版行为只保留三件事：
  - 追击接近：会稳定朝玩家靠近，但不要有突进。
  - 普通攻击：单段横斩或斜砍，前摇清晰，中段短，后摇明显。
  - 受击响应：能吃僵直、击飞、倒地和起身。
- 明确不建议首版加入：
  - 防御
  - 霸体
  - 远程攻击
  - 多段连招
  - 侧移绕背 AI
  - 召唤或地面陷阱
- 体验目标：
  - 玩家单挑时应感觉“稳，但不能乱按”。
  - 玩家面对两只时应自然开始拉距离、等前摇、考虑先后手，而不是无脑冲进去换血。
  - 它打中玩家一次不应致命，但要足够疼，让玩家愿意认真规避。
- 策划上的硬要求：
  - 首怪的攻击前摇必须比玩家当前大多数轻攻击更好读。
  - 首怪的移动速度不要快过玩家步行，更不要快过玩家奔跑。
  - 首怪不能设计成“血厚拖时长”，它的危险应来自反打机会，而不是刮痧。
  - 首怪必须允许玩家用现有长剑的普通攻击链、跑攻、空中攻击、守御反击都能打出明确价值。

## 2026-03-26 敌人动画方案判断
- 结论：第一只正式小怪完全可以使用“逐帧贴图动画”方案，而且从当前工程架构来看，这反而比继续沿用玩家那套类骨骼分件动画更合适。
- 但有一个前提：不要让正式敌人继续继承 `player.tscn + player.gd` 这条链路。

### 为什么可行
- 当前玩家对敌方造成命中，依赖的是“协议接口”而不是“必须共享同一套动画节点”：
  - `receive_weapon_hit()`
  - `receive_grabbed_weapon_hit()`
  - `enter_grabbed_by()`
  - `release_grabbed()`
- 也就是说，只要敌人场景是 `CharacterBody2D`，并实现上述受击/抓取接口，就能接入现有长剑判定系统。
- 现有武器判定与投射物系统本质上关心的是：
  - 目标是不是可碰撞的身体
  - 目标有没有受击接口
  - 命中后能否进入僵直 / 击飞 / 倒地 / 抓取
- 这些都不要求敌人必须使用玩家那种“头 / 身体 / 双腿”的分件动画结构。

### 为什么不建议沿用玩家那套动画结构
- 当前 `player.gd` 对可视节点有很强的硬绑定：
  - `Visuals/BodyPivot/Torso`
  - `Visuals/BodyPivot/HeadPivot`
  - `Visuals/BodyPivot/LeftLegPivot`
  - `Visuals/BodyPivot/RightLegPivot`
  - `AnimationPlayer`
- `reset_visual_pose()` 里直接重置了这些节点的位置和旋转。
- `update_animation_state()` 也直接依赖一整套玩家动作名：
  - `idle / walk / run / jump_up / fall / climb / climb_idle / skid / stun / launch / down / grabbed`
- 训练木桩当前能工作，是因为它本质上就是“继承了玩家整套场景和脚本”，只是把输入关掉了。
- 这条路线适合调试木桩，不适合正式敌人。因为正式敌人的动作语义、轮廓需求、节奏感和资源生产方式都与玩家不同。

### 对第一只怪的推荐实现方式
- 敌人场景独立新建，不继承 `player.tscn`。
- 可视表现使用逐帧贴图：
  - 推荐主节点下放 `AnimatedSprite2D`
  - 或 `AnimationPlayer + Sprite2D.frame` 驱动整张序列帧
- 逻辑层与表现层分离：
  - 逻辑脚本自己维护 `idle / move / attack / stun / launch / down / grabbed / get_up`
  - 表现层只负责切换对应逐帧动画
- 这样做的好处是：
  - 怪物动作更自由，不受玩家人体拆件结构限制
  - 美术产出更直接，适合做“脏、碎、怪、压迫感强”的敌人
  - 第一只怪的开发速度通常会更快

### 当前技术风险
- 如果正式敌人仍然继承 `player.gd`，你后面会遇到三个问题：
  - 需要硬凑出一套玩家式节点层级，徒增美术和场景负担
  - 会被玩家动作状态名绑死，敌人行为越多越别扭
  - 以后每改一次玩家表现脚本，都可能误伤敌人
- 如果正式敌人改走逐帧动画，最大的成本不在技术，而在资源量：
  - 待机
  - 移动
  - 攻击
  - 受击
  - 击飞
  - 倒地
  - 起身
  - 抓取
- 但对第一只怪来说，这是可接受的，因为它本来就应该是“动作少但读招清楚”的基础怪。

### 策划与技术综合建议
- 第一只怪请直接走“独立敌人场景 + 逐帧贴图动画”。
- 不要把它当成“敌方玩家模板”。
- 现阶段最合理的目标不是先做敌我统一动画框架，而是先做出一只：
  - 命中清楚
  - 读招清楚
  - 受击清楚
  - 能稳定接入现有战斗协议
  - 能验证 PVE 节奏
  的正式小怪。

## 2026-03-27 狱庭监工 AI 逻辑
- 狱庭监工不是高速压脸型精英，而是“缓慢推进、重读招、范围压制、带小怪形成主威胁中心”的房间核心单位。
- 它的核心职责：
  - 正面压迫玩家
  - 用大范围重攻击压缩安全区
  - 借由小怪干扰让玩家更难稳定处理其出招空档
- 它的危险不来自复杂机制，而来自“每一下都不能乱吃”。

### 行为状态建议
- `idle`
  - 待机警戒。
  - 玩家未进入感知范围时，原地站定或极短距离缓慢巡逻。
- `approach`
  - 玩家进入感知范围后，缓慢朝玩家推进。
  - 不急跑，不扑脸，不突然突进。
  - 优先保持正面对着玩家。
- `threaten`
  - 进入中距离后短暂停顿，给玩家明确压迫感。
  - 这一状态的作用是让玩家知道它准备出招，而不是无缝贴脸。
- `attack_sweep`
  - 大范围横扫。
  - 用于覆盖正前方和中近距离扇形区域。
- `attack_cleave`
  - 重下劈。
  - 前摇更重，命中惩罚更高。
  - 用于打击在它正前方贪刀或停留过久的玩家。
- `attack_turn_sweep`
  - 慢速回身挥击。
  - 只有当玩家明显绕到它侧后时才使用。
  - 必须给足读招时间，不能瞬发。
- `recovery`
  - 每次攻击后进入明显恢复期。
  - 玩家应能借此找到输出窗口。
- `hit_stun`
  - 被普通攻击打中时进入受击僵直。
  - 不能频繁被小 hit 完全锁死，但也不能完全无反馈。
- `launch`
  - 被强攻击击飞。
- `down`
  - 倒地。
- `get_up`
  - 起身恢复。
- `grabbed`
  - 被抓取或控制状态。

### 战斗逻辑
- 玩家不在感知范围内：
  - `idle`
- 玩家进入感知范围后：
  - 进入 `approach`
- 进入中距离时：
  - 短暂停顿进入 `threaten`
- 根据玩家位置与当前朝向决定攻击：
  - 玩家在正前方中距离：优先 `attack_sweep`
  - 玩家在正前方近距离且停留较久：可选 `attack_cleave`
  - 玩家绕到侧后：使用 `attack_turn_sweep`
- 每次攻击结束后：
  - 必须进入 `recovery`
  - 恢复结束再重新判断 `approach / threaten / attack`

### 行为原则
- 不要连续连招。
- 不要快速取消后摇。
- 不要做突然冲锋。
- 不要高频霸体压人。
- 不要频繁乱转身追镜头。
- 它的强来自“范围和重量”，不是“速度和频率”。

### 与小怪的联动原则
- 当场上有小怪时，狱庭监工不需要更聪明，只需要更稳。
- 它不用抢节奏，而是维持压迫中心：
  - 地牢刀手逼玩家靠后
  - 弩手逼玩家移动
  - 盾卫堵路线
  - 猎犬抓转移空档
  - 监工负责用大范围重攻击收割错误站位

### 玩家应感受到的节奏
- 看到它时：这是房间主威胁。
- 面对它时：不能乱贴脸，必须看它抬手。
- 和小怪一起时：先稳局面，再打精英空档。
- 打赢它时：不是因为背机制，而是因为读懂了它的大动作节奏。

## 2026-03-27 第一只 Boss 设计
- 第一只 Boss 建议做“不动的多部件机关型生命体”，当前推荐名称为“石柱五面像”。
- 核心概念：
  - Boss 本体是一根固定在场中央或场末端的大石柱。
  - 石柱上串联着 5 个人头雕像。
  - 这 5 个头在战斗设计上共同组成一个 Boss，但在技术实现上应视为 5 个独立单位。
  - 每个头都应有自己的场景、自己的生命、自己的受击反馈与自己的技能逻辑。
  - 只有全部头部被击破，Boss 才真正死亡。
- 设计目标：
  - 它不是高速动作 Boss，而是你项目里的第一只“机制阅读型 Boss”。
  - 但它又不能变成纯解谜；玩家仍然要在移动、输出、规避、判断击杀顺序之间做战斗决策。
  - 玩家每处理掉一个头，都应明显感觉到“场面压力被改变了”。

### Boss 的核心体验
- 玩家进入房间后，应立刻理解：
  - 中央的大石柱就是 Boss 本体
  - 5 个头分别在施加不同压力
  - 我不是只要站桩砍柱子，而是要处理机制、找安全窗口、逐个拆掉头
- 战斗节奏应是：
  - 观察当前激活的威胁
  - 处理脚下和场面
  - 找窗口输出一个头
  - 击破一个头后，重新适应新的场面结构

### 5 个头的设计原则
- 每个头都不做“自己冲过来打你”的近战行为。
- 每个头都只通过技能和机制施压。
- 每个头都对应一种非常明确的玩家应对动作。
- 每个头都不能设计成“必须背答案”，而应做到第一次吃亏，第二次就能看懂。
- 每个头的技能应尽量围绕你现有战斗系统支持的东西：
  - 站位
  - 跳跃
  - 守御
  - 移动
  - 输出窗口选择

### 五个头的推荐构成

#### 1. 火头
- 机制定位：延迟场地爆破提示器。
- 地图前提：
  - Boss 房存在左侧平台和右侧平台两个明确区域。
- 技能表现：
  - 火头不会直接喷火打人。
  - 它会先看向左侧或右侧。
  - 眼睛发光，并同步给被选中的平台做明显变色提示。
  - 这代表该侧平台已被“标记为待爆炸区域”。
  - 在火头完成提示后，只要除火头之外的任意一个头成功发动一次技能，就开始进入爆炸倒计时。
  - 倒计时固定为 3 秒，结束后被标记的平台爆炸。
- 逼玩家做的事：
  - 先记住哪边平台被标记。
  - 在其他头继续出招时，提前从对应平台撤离。
  - 不要因为平台还没立刻爆炸就继续贪输出。
- 设计价值：
  - 它不是直接伤害头，而是“战场预告头”。
  - 它把 Boss 战变成带记忆压力的机制战，但可读性仍然很强。

#### 2. 雷头
- 机制定位：大范围预警落雷。
- 技能表现：
  - 随机在地图中的地板或平台表面召唤 10 道雷电预警点。
  - 预警出现 1.5 秒后，雷电落下。
  - 每一道雷电都不是只打落点，而是会贯穿整条垂直线，造成整条纵向伤害。
- 逼玩家做的事：
  - 看预警分布快速换位。
  - 不只看脚下，还要判断自己头顶和脚下是否处在同一雷线中。
  - 学会在左右平台和主地面之间找安全区。
- 设计价值：
  - 它提供最直接的走位考试。
  - 也最能和火头的“平台待爆炸”形成联动压力。

#### 3. 风头
- 机制定位：减速干扰。
- 技能表现：
  - 周期性释放一段风压或风场效果。
  - 被影响到的玩家会进入减速状态，而不是被拖拽或吹飞。
  - 减速持续一小段时间，使玩家更难及时换位和撤离。
- 逼玩家做的事：
  - 提前规划站位。
  - 不要等危险快落下时才开始移动。
  - 学会优先拆掉会让自己“来不及躲”的机制头。
- 设计价值：
  - 它实现简单、规则清楚。
  - 它不直接决定伤害，但会让火头和雷头的威胁显著放大。

#### 4. 守卫头
- 机制定位：反伤惩罚。
- 技能表现：
  - 周期性为除自己之外的其中一个头施加反伤效果。
  - 当玩家攻击这个被标记的头时，对它造成的伤害会同样作用到玩家自己身上。
- 逼玩家做的事：
  - 观察当前哪个头被守卫头保护。
  - 不要在错误时间继续输出被标记的头。
  - 学会切换目标或短暂停手。
- 设计价值：
  - 它把 Boss 战从“只看躲避”提升到“看躲避 + 看输出时机”。
  - 它非常契合你游戏“HP 很珍贵，不能乱吃亏”的核心体验。

#### 5. 恢复头
- 机制定位：可打断治疗。
- 技能表现：
  - 对除自己之外的其中一个头发动持续治疗。
  - 只要恢复头在治疗期间受到玩家伤害，治疗就会立刻被打断。
- 逼玩家做的事：
  - 观察它是否正在读条或持续治疗。
  - 在治疗窗口内优先碰它一下，哪怕不是最高输出选择。
  - 理解“打断”本身就是输出策略的一部分。
- 设计价值：
  - 它定义了这场 Boss 战的“优先处理窗口”。
  - 它让玩家不只是盯着低血头补刀，而是要看场面中谁在改变整体战局。

### 推荐击杀顺序引导
- 默认推荐玩家优先击破顺序：
  - 恢复头
  - 守卫头
  - 火头 / 雷头（二选一，看玩家更怕哪种）
  - 风头
  - 最后剩余的伤害头
- 这样做的原因：
  - 先拆恢复头，避免战斗被拖长。
  - 再拆守卫头，避免自己输出时持续被反伤。
  - 之后场面会明显清爽，玩家会感受到“战斗正在被我拆解”。

### 战斗结构建议
- 不建议五个头从开场同时全功率工作。
- 推荐做成“全体存在，但分层激活”：
  - 开场：火头 + 雷头
  - 血量或时间到阈值后：风头加入
  - 玩家击破任一头后：守卫头或恢复头更频繁介入
- 或更稳妥一点：
  - 五个头都可被攻击
  - 但同一时间最多只有 2 到 3 个头在主动施法
- 这样能避免首个 Boss 房一上来就信息过载。

### 机制联动口径
- 火头的“平台爆炸”不是独立倒计时技能，而是一个“先标记、后由其他头触发”的联动机制。
- 当前正式规则锚定为：
  - 火头先完成左/右平台提示。
  - 之后任意一个非火头成功发动技能。
  - 从该技能发动时开始计时 3 秒。
  - 倒计时结束，被标记的平台爆炸。
- 这条规则非常关键，因为它决定了 Boss 战的阅读顺序是：
  - 先看火头标记了哪边
  - 再看接下来是谁出手
  - 最后预判 3 秒后的平台危险区

### 当前实现友好度判断
- 这五个头里，实现复杂度从低到高大致为：
  - 风头
  - 恢复头
  - 守卫头
  - 雷头
  - 火头
- 其中火头虽然不是直接伤害技能，但由于它带有“平台标记 -> 他头触发 -> 延迟爆炸”的链式关系，实际上是当前最有特色、也最需要先定义清楚事件流的头。

### 场地建议
- 房间应足够宽，允许横向移动和换位。
- 不建议做多层复杂平台。
- 第一只 Boss 的场地应更像“压迫式长平台”或“宽平台 + 少量高低差”。
- 重点是让玩家看清机制，不是让场地本身变成主要难点。

### 和小怪的关系
- 第一只 Boss 不建议一开场就加小怪。
- 更建议做两种方式之一：
  - Boss 单体纯机制战
  - 某个头被击破后，短暂召入 1 到 2 只低压小怪作为干扰
- 当前更推荐第一种：
  - 先把这只多部件 Boss 的机制阅读做好
  - 后续第二只 Boss 再考虑 Boss + 小怪常态化组合

### 玩家体验目标
- 第一次打：
  - 看不懂会吃亏，但能看见自己为什么吃亏
- 第二次打：
  - 开始知道先打哪个头
  - 开始学会留输出窗口
- 熟练后：
  - 会形成自己的拆头路线
  - 会感觉这是一场“我在拆解一个危险机关”的战斗，而不是纯拼反应

### 当前策划建议总结
- 这只 Boss 的重点不是动作表演，而是“多机制并存下的优先级战斗”。
- 它最适合做你游戏里的第一只 Boss，因为它能把：
  - 稀缺 HP
  - 小心推进
  - 读机制
  - 输出选择
  这几条核心体验收束到一场战斗里。
